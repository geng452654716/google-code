- 产品暂定名：Google Code
- 产品形态：桌面应用
- 目标用户：个人自用
- 核心能力：本地、离线、安全地管理账号并生成 TOTP 动态验证码

> 命名说明：“Google Code”当前作为项目代号使用。若未来公开发布，应避免让用户误认为本产品由 Google 官方提供，并重新评估产品名称、图标及商标表述。

## 1. 产品概述

Google Code 是一款个人使用的桌面 TOTP 动态验证码生成器，功能类似 Google Authenticator。

用户可以通过手动输入、`otpauth://` 链接、二维码图片、摄像头、系统截图以及 Google Authenticator 导出二维码等方式导入账号。应用在本机保存账号数据，不要求登录，不依赖云端服务，并能够在离线状态下持续生成动态验证码。

产品重点解决以下问题：

1. 在桌面端快速查看和复制验证码，减少手机与电脑之间来回切换。
2. 支持多种二维码输入方式，尤其是剪贴板粘贴和屏幕截图识别。
3. 将敏感密钥安全地保存在当前设备，不上传服务器。
4. 为大量账号提供搜索、分组、排序、批量导入和备份能力。

## 2. 产品目标

### 2.1 核心目标

- 正确生成符合 TOTP 标准的动态验证码。
- 常见场景下，用户能在 3 秒内找到并复制目标验证码。
- 覆盖桌面环境中常用的账号导入方式。
- 所有账号密钥只在当前设备本地保存和处理。
- 支持应用锁、系统生物识别和加密备份，降低密钥泄露风险。

### 2.2 成功标准

- 常见 `otpauth://` 数据可以被正确解析和生成验证码。
- 二维码文件、剪贴板图片、截图区域和摄像头均能完成二维码识别。
- 应用完全断网时，已有账号仍可正常显示验证码。
- 应用退出并重新打开后，账号、分组和排序结果保持不变。
- 锁定状态下不显示账号密钥和验证码。
- 数据导出后可以在新安装的应用中完整恢复。

### 2.3 非目标

首个版本暂不包含：

- 用户注册、登录和云端同步。
- 多设备实时同步。
- 团队共享、权限管理和审计后台。
- 短信验证码、邮箱验证码和验证码代收。
- Google OAuth 授权码或访问令牌管理。
- 自动登录第三方网站。
- 浏览器扩展和手机 App。
- HOTP 计数器验证码，除非后续确认需要。

## 3. 用户画像与使用场景

### 3.1 用户画像

- 单一用户，主要在自己的桌面电脑上使用。
- 拥有多个开启两步验证的账号。
- 重视操作效率，但不希望数据上传云端。
- 可能需要从 Google Authenticator 一次性迁移多个账号。

### 3.2 典型场景

#### 场景 A：快速复制验证码

1. 用户打开应用或使用快捷入口唤起应用。
2. 搜索账号名称。
3. 点击验证码或复制按钮。
4. 验证码写入剪贴板。
5. 应用提示复制成功，并在设定时间后自动清除剪贴板中的验证码。

#### 场景 B：粘贴二维码截图

1. 用户在网页或其他应用中复制二维码图片。
2. 切换到 Google Code。
3. 使用粘贴操作或点击“从剪贴板导入”。
4. 应用识别二维码并展示待导入账号。
5. 用户确认名称、发行方和分组后保存。

#### 场景 C：调用系统截图扫描

1. 用户点击“截取屏幕二维码”。
2. 应用调用系统截图或区域选择能力。
3. 用户框选其他窗口中的二维码。
4. 应用识别截图中的二维码。
5. 用户确认并保存账号。

#### 场景 D：从 Google Authenticator 批量迁移

1. 用户在 Google Authenticator 中导出账号二维码。
2. 在桌面应用中通过摄像头、图片、剪贴板或截图读取二维码。
3. 应用解析迁移数据；如果存在多张迁移二维码，引导用户继续扫描。
4. 应用展示全部待导入账号和重复项。
5. 用户批量确认并完成导入。

#### 场景 E：本地备份与恢复

1. 用户选择导出加密备份。
2. 设置或确认备份密码。
3. 应用生成本地备份文件。
4. 用户在需要恢复时选择该文件并输入密码。
5. 应用展示恢复预览、冲突处理方式和最终结果。

## 5. 详细功能需求

## 5.1 首次启动与安全设置

### 5.1.1 首次启动

首次启动时，应用应：

1. 简要说明产品用途及“数据仅保存在本机”。
2. 引导用户设置应用主密码或本地解锁 PIN。
3. 如果操作系统支持，询问是否启用系统生物识别解锁。
4. 允许用户选择界面语言和主题，也可直接使用系统设置。
5. 创建空账号库并进入首页。

### 5.1.2 应用锁

- 应用启动时默认进入锁定状态，具体策略可在设置中调整。
- 支持主密码或 PIN 解锁。
- 支持调用系统安全能力进行生物识别解锁。
- 支持以下自动锁定条件：
  - 应用进入后台或窗口失焦后立即锁定。
  - 离开应用达到指定时间后锁定。
  - 电脑锁屏或休眠后锁定。
  - 用户手动锁定。
- 连续多次解锁失败时增加等待时间，避免暴力尝试。
- 忘记主密码时不提供绕过机制；用户只能清空本地数据后重新开始，或通过备份恢复。

## 5.2 验证码首页

首页以账号列表为核心，每个账号至少展示：

- 账号名称，例如邮箱或用户名。
- 发行方/服务名称，例如 GitHub、Google。
- 当前六位验证码。
- 当前周期剩余秒数或环形倒计时。
- 复制操作。
- 所属分组或分组入口。

行为要求：

- 默认按照用户设置的顺序显示账号。
- 验证码到期后自动刷新，不要求用户手动操作。
- 临近过期时应通过颜色或动画提示，但不能影响可读性。
- 用户点击验证码或复制按钮即可复制。
- 复制成功后展示轻量反馈。
- 如果验证码即将在极短时间内过期，应用应提示用户等待下一组或在复制后明确剩余时间。
- 应用被锁定后，首页内容必须立即隐藏。

## 5.3 搜索、分组和排序

### 5.3.1 搜索

- 支持按账号名称和发行方进行不区分大小写的模糊搜索。
- 搜索结果随输入实时更新。
- 无结果时提供“添加账号”入口。
- 可选支持快捷键聚焦搜索框。

### 5.3.2 分组

- 用户可以新建、重命名和删除分组。
- 一个账号默认属于一个分组；是否支持多分组归属后续确认。
- 删除分组时，分组内账号默认移至“未分组”，不得静默删除账号。
- 首页可以按分组筛选或折叠展示。

### 5.3.3 排序

- 支持手动拖拽排序。
- 支持按账号名称、发行方和最近使用时间排序。
- 支持置顶常用账号。
- 排序结果应本地持久化。

## 5.4 添加账号

所有导入方式最终都进入统一的“导入确认”页面。确认页显示解析结果，允许用户修改非密钥信息，并在保存前发现重复项或错误。

### 5.4.1 手动输入

字段包括：

- 发行方，可选。
- 账号名称，必填。
- Secret 密钥，必填。
- 分组，可选。
- 算法，默认 SHA-1。
- 位数，默认 6 位。
- 周期，默认 30 秒。

要求：

- 自动去除密钥中的空格和常见分隔符。
- 对 Base32 密钥进行格式校验。
- 保存前可以预览当前验证码。
- Secret 输入框默认隐藏，并提供临时显示功能。

### 5.4.2 粘贴 `otpauth://` 链接

- 支持解析标准 TOTP URI。
- 解析账号、发行方、Secret、算法、位数和周期。
- URI 缺少必填参数或参数无效时，明确指出问题。
- 不应在普通日志中记录完整 URI 或 Secret。

### 5.4.3 上传二维码图片

- 支持从本地选择常见图片格式。
- 一张图片中存在一个二维码时直接解析。
- 一张图片中存在多个二维码时列出识别结果供用户选择。
- 图片无法识别时提供清晰错误信息和重试建议。
- 图片只在本机内存或临时目录中处理，完成后及时清理临时文件。

### 5.4.4 剪贴板图片粘贴

- 支持 `Ctrl+V` / `Cmd+V` 直接粘贴图片。
- 支持点击“从剪贴板导入”。
- 剪贴板不是图片时，不得错误覆盖已有账号数据。
- 识别完成后进入统一确认页。

### 5.4.5 系统截图扫描

- 提供“扫描屏幕二维码”入口。
- 调用操作系统的区域截图能力，允许用户框选屏幕区域。
- 首次使用时，引导用户授予必要的屏幕录制或截图权限。
- 启动系统截图前明确说明窗口会暂时离开屏幕、鼠标会变成十字光标，并提示 `Esc` 可以取消，避免把正常截图交互误认为应用闪退。
- 成功、取消和异常结束后都必须恢复并激活应用窗口；取消时显示非错误状态提示，不产生错误账号。
- 截图完成后自动识别，并进入确认页。
- 截图只用于当前识别任务，默认不长期保存。

### 5.4.6 摄像头扫描

- 列出可用摄像头并允许切换。
- 首次使用时请求摄像头权限并解释用途。
- 实时识别二维码，识别成功后暂停扫描，避免重复导入。
- 关闭扫描页面后释放摄像头资源。
- 权限被拒绝时提供进入系统设置的说明。

### 5.4.7 Google Authenticator 导出二维码

- 能识别 Google Authenticator 迁移二维码中的一个或多个账号。
- 支持同一批次包含多张二维码的情况，并显示扫描进度。
- 批量导入前展示账号列表。
- 对重复账号、无效数据和已存在账号分别标记。
- 允许用户取消勾选不需要导入的账号。
- 不完整批次不得在无提示的情况下当作完整导入。

## 5.5 导入确认与重复处理

导入确认页面应展示：

- 发行方。
- 账号名称。
- 验证码类型。
- 算法、位数和周期。
- 分组。
- 重复状态或异常提示。

重复账号建议以“规范化后的发行方 + 账号名称 + Secret”判断，并支持：

- 跳过重复项。
- 更新已有账号的展示信息。
- 作为新账号保留。

任何批量操作完成后，应展示成功、跳过和失败数量。

## 5.6 编辑与删除账号

### 5.6.1 编辑

- 可以修改发行方、账号名称、分组、图标和排序等展示信息。
- 修改 Secret、算法、位数或周期时应二次确认，因为会改变验证码结果。
- 编辑页面不应默认明文展示 Secret。

### 5.6.2 删除

- 删除前必须确认。
- 批量删除需要再次确认账号数量。
- 可选提供本地回收站或短时间撤销能力。
- 删除后应清除对应敏感数据，不能仅从列表隐藏。

## 5.7 导出、备份与恢复

### 5.7.1 默认导出方式

默认提供应用专用的加密备份文件，备份至少包含：

- 账号及 TOTP 参数。
- 分组。
- 排序和置顶信息。
- 与账号相关的非系统设置。
- 备份格式版本。

要求：

- 导出时必须设置或确认备份密码。
- 备份密码与应用解锁密码可以不同。
- 备份文件不能以明文形式存储 Secret。
- 导出成功后明确显示保存位置。
- 不在日志、最近操作记录或通知中泄露密码和 Secret。

### 5.7.2 恢复

- 选择备份文件并输入密码后，先校验文件完整性。
- 恢复前显示内容预览和账号数量。
- 支持“合并到当前数据”和“替换当前数据”。
- 替换模式必须二次确认，并建议先备份当前数据。
- 支持重复项处理和部分失败报告。
- 无法识别的未来版本或损坏文件需要明确提示，不得破坏当前数据。

### 5.7.3 兼容性导出

是否提供以下能力需要确认：

- 导出标准 `otpauth://` 数据。
- 生成可供其他认证器扫描的二维码。
- 生成兼容 Google Authenticator 的批量迁移二维码。
- 明文 CSV/JSON 导出。

若提供明文或二维码导出，必须展示高风险警告，并要求再次验证用户身份。

## 5.8 单个账号分享

用户可以从某个账号的操作菜单中选择“分享账号”，将该账号迁移给其他认证器或设备。分享的是能够持续生成验证码的原始凭据，不是仅分享当前一次性的六位验证码。

### 5.8.1 分享内容

支持按需提供以下形式：

- 显示并复制原始 Base32 Secret。
- 复制标准 `otpauth://` 链接。
- 生成包含完整 `otpauth://` 数据的二维码，供其他认证器扫描。
- 将二维码保存为本地图片。
- 调用操作系统分享面板分享二维码图片或文本，平台支持时启用。

二维码及链接应包含该账号的发行方、账号名称、Secret、算法、位数和周期，确保接收方导入后能生成一致的验证码。

### 5.8.2 分享流程

1. 用户在账号菜单中点击“分享账号”。
2. 应用明确警告：获得 Secret 或二维码的人可以持续生成该账号的验证码。
3. 用户必须重新输入主密码、PIN，或完成一次系统生物识别。
4. 用户选择分享形式：Secret、链接或二维码。
5. 应用展示分享内容并允许复制、保存或调用系统分享面板。
6. 分享完成、窗口失焦或达到超时时间后，应用自动隐藏敏感内容。

### 5.8.3 分享安全限制

- 不允许从锁定状态直接分享。
- 每次分享都必须重新验证身份，不能仅依赖当前已解锁状态。
- 分享确认页不得默认直接显示完整 Secret。
- 分享二维码和 Secret 应带有明显的高风险标识。
- 分享页进入后台、窗口失焦或系统锁屏时应立即遮挡敏感信息。
- 二维码图片默认不写入应用长期缓存；临时文件使用完成后应清理。
- 调用系统分享面板前，应再次说明数据将离开本应用控制范围。
- 不在日志、通知、最近操作或崩溃信息中记录分享内容。
- 复制 Secret 或 `otpauth://` 链接后，应使用敏感剪贴板自动清除策略。
- 首版仅支持单个账号分享；批量分享继续通过加密备份或后续迁移二维码实现。

## 5.9 剪贴板安全

- 复制验证码后，可配置在 10、30 或 60 秒后自动清除。
- 清除前需要确认剪贴板内容仍是本应用写入的验证码，避免覆盖用户后来复制的内容。
- 默认不自动复制 Secret。
- 复制 Secret 时必须二次确认，并采用更短的自动清除时间。

## 5.10 设置

设置项包括：

- 语言：跟随系统、简体中文、英文。
- 主题：跟随系统、浅色、深色。
- 自动锁定策略和时长。
- 启动时是否立即锁定。
- 生物识别开关。
- 剪贴板自动清除时长。
- 默认排序方式。
- 验证码临近过期提醒阈值。
- 是否开机启动，P2。
- 全局快捷键，P2。
- 数据备份、恢复和清空。
- 当前版本和隐私说明。

## 6. 页面与信息架构

建议首版包含以下页面或窗口：

1. **首次启动引导**：产品说明、安全设置、语言和主题。
2. **解锁页**：主密码/PIN、生物识别、清空数据入口。
3. **验证码首页**：搜索、分组、账号列表、倒计时、复制。
4. **添加账号菜单**：展示全部导入方式。
5. **手动添加页**：填写 TOTP 参数。
6. **二维码导入页**：图片、剪贴板、截图、摄像头入口。
7. **导入确认页**：结果预览、编辑、重复处理、批量保存。
8. **账号详情/编辑页**：修改信息、查看参数、分享、删除。
9. **账号分享确认页**：身份验证、安全警告、Secret、链接和二维码分享。
10. **分组管理页**：新增、编辑、删除、排序。
11. **备份与恢复页**：加密导出、恢复预览和结果。
12. **设置页**：安全、剪贴板、外观、语言和数据管理。

## 7. 数据模型建议

### 7.1 Account

| 字段 | 说明 |
| --- | --- |
| `id` | 本地唯一标识 |
| `issuer` | 发行方/服务名称 |
| `accountName` | 账号名称 |
| `secretEncrypted` | 加密后的 TOTP Secret |
| `algorithm` | SHA-1、SHA-256 或 SHA-512 |
| `digits` | 默认 6 |
| `period` | 默认 30 秒 |
| `groupId` | 所属分组 |
| `sortOrder` | 手动排序值 |
| `isPinned` | 是否置顶 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |
| `lastUsedAt` | 最近复制或使用时间 |

### 7.2 Group

| 字段 | 说明 |
| --- | --- |
| `id` | 本地唯一标识 |
| `name` | 分组名称 |
| `sortOrder` | 排序值 |
| `createdAt` | 创建时间 |
| `updatedAt` | 更新时间 |

### 7.3 LocalSettings

保存语言、主题、锁定策略、剪贴板策略和默认排序等非敏感设置。加密密钥、主密码派生结果和生物识别授权信息应使用操作系统安全存储，不应直接写入普通配置文件。

## 8. 安全与隐私要求

### 8.1 数据边界

- 不要求网络连接。
- 不提供云端账号体系。
- 不上传账号、Secret、二维码、截图、摄像头画面或使用行为。
- 若未来加入自动更新或崩溃上报，必须单独说明联网行为，并默认不包含敏感信息。

### 8.2 本地存储

- Secret 必须加密后持久化。
- 优先使用操作系统 Keychain、Credential Manager 等安全存储保护主密钥。
- 主密码不得明文保存。
- 数据库、缓存、临时文件和日志中不得出现明文 Secret。
- 应用锁定后，应尽量释放内存中的解密数据。

### 8.3 日志

日志中禁止记录：

- 完整 Secret。
- 完整 `otpauth://` URI。
- 当前或历史验证码。
- 主密码、PIN、备份密码。
- 未脱敏的二维码解析原文。

### 8.4 屏幕与窗口

- 锁定时隐藏全部账号信息和验证码。
- 应用切入后台或进入系统任务切换视图时，可根据平台能力遮挡敏感内容。
- 通知中不得展示 Secret；默认也不展示完整验证码。
- 分享 Secret 或二维码时，窗口失焦、进入后台或超时后应自动遮挡。

## 9. 非功能需求

### 9.1 准确性

- TOTP 生成应符合 RFC 6238。
- Base32 解析和 URI 解析应覆盖常见兼容格式。
- 使用系统时间生成验证码。
- 当系统时间明显异常时提示用户校准，不自行联网修改时间。

### 9.2 性能

- 在常用桌面设备上冷启动目标不超过 3 秒。
- 解锁后账号列表目标在 1 秒内可交互。
- 1000 个账号规模下，搜索输入和验证码刷新不应产生明显卡顿。
- 截图或普通二维码图片的识别目标在 2 秒内完成。

### 9.3 稳定性

- 单个二维码解析失败不能导致应用崩溃。
- 批量导入应具备事务性：全部成功提交，或明确报告部分成功状态。
- 数据写入过程中异常退出，不应损坏整个账号库。
- 恢复备份失败时应保留恢复前数据。

### 9.4 可用性

- 支持键盘操作。
- 关键按钮有清晰文字或辅助说明。
- 深色模式下保证验证码和倒计时具有足够对比度。
- 中英文切换后不需要重启，或明确提示重启范围。

### 9.5 平台兼容

目标平台尚待确认。需求层面预留：

- macOS：系统截图权限、摄像头权限、Keychain、Touch ID。
- Windows：屏幕捕获、摄像头权限、Credential Manager、Windows Hello。
- Linux：不同桌面环境下的截图、密钥环和生物识别能力可能不统一，可后置支持。

### 9.6 个人安装包

- macOS 提供包含 `.app` 和 `Applications` 快捷方式的 DMG，并附 SHA-256。
- Windows 提供当前用户级 Setup EXE，并附 SHA-256；不要求管理员权限。
- 安装包允许覆盖升级，并在卸载时保留 Vault、系统安全存储记录和用户备份。
- 当前个人安装包不绕过 Gatekeeper/SmartScreen，不宣称具备 Developer ID、公证或 Authenticode 身份认证。
- CI 可以生成短期私有 Artifact，但不得自动创建公开 GitHub Release。

## 10. 异常与边界场景

应用需要覆盖以下情况：

- 二维码清晰度过低、反色、旋转或部分遮挡。
- 图片中没有二维码或存在多个二维码。
- 二维码内容不是 `otpauth://` 或不受支持的迁移数据。
- URI 缺少 Secret、Secret 非法、周期或位数不合法。
- 同一账号被重复导入。
- Google Authenticator 多二维码迁移批次缺失其中一张。
- 摄像头或截图权限被拒绝。
- 系统没有摄像头。
- 剪贴板中没有图片。
- 系统时间错误导致验证码与服务端不一致。
- 主密码错误、备份密码错误、备份文件损坏。
- 磁盘空间不足或本地数据库不可写。
- 应用升级后本地数据结构迁移失败。
- 分享过程中用户取消系统分享面板、目标应用不可用或二维码保存失败。
- 分享页面停留过久、窗口失焦或系统锁屏。

## 11. P0 MVP 验收标准

首个可运行版本满足以下条件即可视为 P0 完成：

1. 用户可以设置本地解锁密码并进入应用。
2. 用户可以手动输入 Base32 Secret 添加 TOTP 账号。
3. 用户可以粘贴 `otpauth://` URI 添加账号。
4. 用户可以选择本地二维码图片进行识别和导入。
5. 用户可以直接粘贴剪贴板中的二维码图片。
6. 用户可以调用系统区域截图，并识别截图中的二维码。
7. 首页正确显示账号、六位验证码和倒计时。
8. 验证码到期后自动刷新。
9. 用户可以一键复制验证码，并收到成功提示。
10. 用户可以搜索、编辑和删除账号。
11. 账号数据在退出应用后仍保留，且 Secret 不以明文存储。
12. 应用锁定后不显示账号和验证码。
13. 在断网环境中，以上核心能力均可正常使用。
14. 中文界面下不存在阻塞流程的未翻译文本。
15. 核心解析、TOTP 计算、加密存储和数据迁移具备自动化测试。
16. macOS DMG 和 Windows Setup EXE 可以在对应平台构建、校验、安装、重复升级和卸载，且不会删除外部 Vault fixture。

## 12. 分阶段交付建议

### 阶段 1：P0 核心闭环

- 应用骨架和本地数据库。
- 应用锁和本地加密。
- 账号 CRUD。
- 手动输入和 URI 导入。
- 图片、剪贴板、截图二维码识别。
- 验证码列表、倒计时、复制和搜索。
- 基础中文界面。

### 阶段 2：完整导入和管理

- 摄像头扫描。
- Google Authenticator 批量迁移二维码。
- 分组、排序、置顶。
- 加密备份与恢复。
- 单个账号的 Secret、`otpauth://` 链接和二维码安全分享。
- 生物识别。

### 阶段 3：体验增强

- 深色模式和英文。
- 托盘、全局快捷键和快速窗口。
- 更多导出兼容格式。
- 性能、可访问性和跨平台完善。

## 13. 风险与注意事项

1. **主密码恢复风险**：纯本地加密意味着忘记主密码后无法找回，必须在首次设置时明确告知。
2. **系统时间风险**：TOTP 完全依赖设备时间，时间偏差会导致验证码无效。
3. **截图权限差异**：macOS、Windows 和 Linux 的授权流程及 API 不同，应先确定首发平台。
4. **迁移二维码兼容性**：Google Authenticator 批量迁移格式与标准 `otpauth://` 二维码不同，需要单独解析和测试。
5. **备份泄露风险**：备份包含可恢复的 Secret，必须强加密并清楚提示用户保管责任。
6. **产品命名风险**：公开发布时使用 Google 品牌相关名称可能造成官方关联误解。
7. **跨平台安全能力差异**：生物识别、安全存储和屏幕保护能力无法保证完全一致。
8. **账号分享风险**：Secret 和二维码不是一次性信息，任何接收者都能长期生成验证码；必须通过重新验证、安全警告和自动遮挡降低误分享风险。

## 14. 本轮待确认事项

以下问题不会阻塞 PRD 初稿，但会直接影响技术选型和版本排期：

1. **首发平台**：只做 macOS、只做 Windows，还是 macOS + Windows？Linux 是否需要？
2. **技术偏好**：是否接受 Flutter Desktop、Tauri、Electron，还是由技术方案阶段推荐？
3. **应用锁策略**：首次版本必须使用独立主密码，还是优先仅使用系统登录认证/生物识别？
4. **批量导出格式**：只需要应用自己的加密备份，还是必须兼容 Google Authenticator 导入二维码？
5. **验证码参数**：产品严格只支持 6 位、30 秒、SHA-1，还是兼容 URI 中的 8 位、其他周期和 SHA-256/SHA-512？
6. **快捷入口**：托盘图标、全局快捷键和“置顶小窗口”是否属于首版刚需？
7. **分组规则**：一个账号只属于一个分组，还是可以同时属于多个分组？
8. **截图体验**：希望应用自己实现区域截图浮层，还是调用系统截图工具后读取剪贴板也可以接受？
9. **明文导出**：是否完全禁止 CSV/JSON 等明文导出？建议默认禁止。
10. **界面方向**：更偏向 Google Authenticator 的简洁列表，还是 1Password/Authy 一类带侧边栏和分组的桌面布局？
11. **分享优先级**：单个账号分享应属于首版 P0，还是按当前建议放在 P1？
12. **分享渠道**：只需复制 Secret、显示二维码和保存图片，还是必须同时调用系统分享面板？
