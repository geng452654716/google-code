# Phase 26：新增 GA 后 GitHub 自动加密备份

日期：2026-07-21

## 阶段目标

在已完成并经用户实测可用的 GitHub 私有仓库备份基础上，增加可选的“新增账号后自动备份”。本阶段保持本地 Vault 为主事务，不因网络或 GitHub 失败丢弃已经保存的 GA。

## 已完成

### 1. 设备本地自动备份配置

- GitHub 已连接并选择私有仓库后显示自动备份开关；
- 功能默认关闭，必须由用户主动开启；
- 开启时要求输入并确认至少 8 个字符的独立备份密码；
- 密码仅保存到当前设备的 Keychain（macOS）或 Credential Manager（Windows）；
- 密码不进入 Vault、`.gcbak`、GitHub、日志或普通配置文件；
- 开启时立即上传当前 Vault，只有验证上传成功后才保持开启；
- 初次上传失败会删除刚写入的设备密码并回滚为关闭状态。

### 2. 新增账号后触发

自动备份只在以下本地保存成功后触发：

- 手动添加单个账号；
- `otpauth://` 导入；
- 图片、截图、剪贴板和摄像头二维码导入；
- Google Authenticator 迁移二维码批量导入。

`addAccounts` 的一次批量保存只触发一次云备份。编辑、删除、创建或调整分组、修改偏好、复制验证码和恢复备份不会触发。

### 3. 失败隔离与并发控制

- 账号先加密保存到本地，再异步执行 GitHub 自动备份；
- GitHub 失败不会回滚或删除本地账号；
- 成功和失败都使用不包含 Token、密码、Secret、验证码或 Vault 明文的提示；
- 快速连续新增时不并发上传，当前上传结束后只继续上传最新待处理 Vault，避免 GitHub `sha` 更新冲突；
- 自动备份密码读取和写入使用临时字节缓冲区，使用后清零。

### 4. 关闭与断开

- 关闭自动备份：删除当前设备的自动备份密码，保留 GitHub 授权和仓库中的加密文件；
- 断开 GitHub：同时关闭自动备份，删除当前设备的 GitHub session 和自动备份密码；
- 两种操作都不会删除 GitHub 私有仓库中的 `.gcbak`。

## 安全边界

- 自动备份配置属于设备本地状态，不写入 `VaultPayload.preferences`；
- 新电脑不会从备份中获得 GitHub Token、自动备份开关或备份密码；
- 新电脑恢复时需要重新连接 GitHub，并输入用户记住的独立备份密码；
- 首版只支持 GitHub 自动备份。iCloud Drive 和 Google Drive 当前依赖用户选择同步目录，没有持久安全目录授权，因此继续保持手动备份。

## 自动化验证

新增或更新测试覆盖：

- 开启时立即验证上传并写入设备安全存储；
- 验证上传失败时删除密码并保持关闭；
- 连续新增串行上传并最终保存最新 Vault；
- 应用启动配置校验尚未完成时，新增账号会等待校验，不会漏掉首次自动备份；
- 单个新增和批量新增各触发一次；
- 编辑和删除账号不触发；
- GitHub 卡片显示自动备份开关；
- 自动备份密码弹窗执行最小长度校验。

## 当前限制

- 自动备份只在应用运行且 Vault 已成功新增账号后触发，不是后台定时服务；
- 网络中断或 GitHub 暂时失败时会提示，但首版不做磁盘持久化重试队列；下次新增账号会再次尝试上传最新 Vault；
- Windows 仍需在 Windows 10/11 真机验证 Credential Manager、GitHub 网络访问和新增后自动上传。
