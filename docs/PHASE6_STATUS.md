# 阶段 6 / Google Authenticator 迁移二维码批量导入状态

- 更新时间：2026-07-16
- 当前结论：已完成 `otpauth-migration://offline` 迁移二维码识别、Protobuf wire 解码、多账号映射、多张二维码批次收集、进度提示、批量勾选、重复标记、无效项汇总和单次 Vault 加密保存闭环。

## 本阶段已完成

### 迁移协议解析

- [x] 识别 `otpauth-migration://offline?data=...`，并与标准 `otpauth://totp` 共用同一导入分类入口。
- [x] 对 Base64URL 和 Protobuf payload 设置编码长度、解码字节、账号数量、Secret、文本和批次数量上限。
- [x] 使用边界明确的只读 Protobuf wire reader 解析字段，不引入网络服务，也不长期保留原始 URI 或完整 payload。
- [x] 显式映射 SHA-1、SHA-256、SHA-512、6 位、8 位和 TOTP；HOTP、MD5、未知算法、未知位数和空数据标记为无效项，不静默降级。
- [x] Secret 字节直接转换为无 padding 的 RFC 4648 Base32，并继续复用现有账号规范化和 Vault 校验。
- [x] 兼容 `issuer:account` 名称形式，避免发行方同时存在于名称和 issuer 字段时重复显示。

### 多二维码批次

- [x] 按 batch ID、batch size 和 batch index 在已解锁内存中收集同一迁移批次。
- [x] 支持任意顺序读取批次二维码，并忽略同一 index 的重复扫描。
- [x] 扫描到其他批次时不覆盖当前未完成批次，并提示继续当前批次或先取消。
- [x] 未完成时显示 `已读取 N/M 张`，可继续选择图片或继续区域截图。
- [x] 未完整收集前不展示“完整导入”确认，也不会写入 Vault。
- [x] 用户取消批次后清除页面持有的迁移条目和 Secret 引用。

### 批量确认与保存

- [x] 完整批次进入独立确认对话框，展示发行方、账号名称、算法、位数、周期和状态，不显示 Secret。
- [x] 有效且非重复账号默认勾选；已存在账号和批次内重复项默认不勾选，但用户可明确选择保留副本。
- [x] 无效项禁用选择并展示安全原因，例如 HOTP 暂不支持。
- [x] 支持选择全部有效账号、全部取消和逐项勾选。
- [x] 所选账号通过 `addAccounts` 一次性规范化并在一个 Vault 保存事务中写入，避免逐条保存形成中间状态。
- [x] 完成后汇总成功、跳过和无效数量。
- [x] 图片、区域截图和剪贴板二维码入口均可识别迁移 payload；未完成批次可跨入口继续收集。

## 当前限制与风险

- [ ] 当前 fixtures 由测试内编码器生成；仍需使用 Android/iOS 最新 Google Authenticator 的真实导出二维码做兼容性回归。
- [ ] 同一张图片或截图中存在多个独立二维码时，当前 QR 解码器仍只返回首个可识别二维码；同一迁移批次的多张码需要逐张选择或截图。
- [ ] 当前重复处理支持“跳过”或“明确保留副本”，尚未提供用迁移项更新既有账号展示信息的独立操作。
- [ ] 未完成批次只保存在当前页面内存中；锁定应用、关闭页面或退出应用后不会恢复扫描进度。
- [ ] 尚未实现摄像头连续扫描，因此大量迁移二维码当前优先通过图片或区域截图逐张读取。
- [ ] Google Authenticator 迁移格式没有纳入应用自身备份格式；本应用导出仍应使用后续的加密备份方案。
