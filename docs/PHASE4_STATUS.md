# 阶段 4 / macOS 区域截图二维码导入状态

- 更新时间：2026-07-16
- 当前结论：已完成 macOS 区域截图、屏幕录制权限引导、取消处理、窗口恢复、二维码识别与 Vault 加密保存闭环；Dart 流程和 macOS Debug/Release 构建已验证。Windows 自有区域选择器已在后续 [阶段 5](./PHASE5_STATUS.md) 实现，但仍待 Windows 环境编译和运行验收。

## 本阶段已完成

### 截图平台边界

- [x] 建立可测试的 `ScreenCaptureService`，将界面流程与原生截图实现隔离。
- [x] 使用 `Uint8List?` 表达截图结果：图片字节表示成功，`null` 表示用户取消。
- [x] 将原生失败归类为权限拒绝、平台不可用和截图失败，不向界面暴露原生诊断或敏感内容。
- [x] 截图图片只通过内存和系统剪贴板传递，不创建应用临时截图文件，也不上传网络。
- [x] Windows runner 已注册同名平台通道；后续阶段 5 已将其连接到自有区域选择器。

### macOS 原生截图

- [x] 使用 `CGPreflightScreenCaptureAccess` 检查屏幕录制权限，并通过 `CGRequestScreenCaptureAccess` 发起系统授权。
- [x] 权限未授予时显示说明对话框，并支持一键打开“隐私与安全性 > 屏幕录制”设置。
- [x] 调用系统 `screencapture -i -s -c -x` 区域选择器，强制区域模式并关闭截图声音。
- [x] 启动选择器前隐藏主窗口，在成功、取消和执行失败路径中恢复并激活应用窗口。
- [x] 使用 `NSPasteboard.changeCount` 区分用户取消与成功截图。
- [x] 复用剪贴板 PNG/TIFF/`NSImage` 内存读取逻辑，避免重复实现图片转换。

### 应用导入流程

- [x] 添加账号菜单新增“区域截图扫描”，说明为“框选屏幕区域并识别二维码”。
- [x] 截图成功后复用现有图片大小限制、QR 预处理、TOTP URI 校验和错误提示。
- [x] 识别结果继续进入统一编辑确认、重复检测和 Vault 加密保存流程。
- [x] 导入来源显示为“区域截图”，不会展示截图像素、剪贴板内容或原始 URI。
- [x] 用户按 Esc 取消区域选择时静默结束，不保存账号、不显示错误。
- [x] 所有成功和失败路径都会复位导入中状态，防止按钮长期不可用。

## 当前限制与风险

- [ ] macOS 应用启用了 App Sandbox；系统 `screencapture` 子进程、TCC 授权后的首次使用和多显示器框选仍需在真实发布形态人工验证，构建成功不能替代运行时验证。
- [ ] macOS 用户首次授权后，系统可能要求重新打开应用；界面已给出相应提示，但尚未自动重启应用。
- [x] Windows 自有区域选择遮罩已在阶段 5 实现；仍未在 Windows 环境编译和实测。
- [ ] 同一张截图中的多个二维码仍只读取首个可识别内容。
- [x] Google Authenticator 导出/迁移二维码批量协议已在阶段 6 实现。
- [ ] 摄像头扫描、安全分享、生物识别和加密备份恢复 UI 尚未实现。
